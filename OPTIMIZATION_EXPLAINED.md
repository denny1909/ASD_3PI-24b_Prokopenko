# ⚡ ОПТИМІЗАЦІЯ: Що змінилося та чому

## 🐌 Була проблема:

Для **1000 точок**:
- Кількість можливих комбінацій: C(1000,3) = **166,167,000 трикутників**
- Старий алгоритм намагався **ЗБЕРЕГТИ ВСІ** ці трикутники в пам'яті
- Пам'ять: ~166 млн × 100 байт = **~16 ГБ**
- Час: **5+ хвилин** (через постійні збої пам'яті)

## ✨ Що зроблено:

### 1. **PriorityQueue замість ArrayList**
```java
// БУЛО (старий код):
List<Triangle> allTriangles = new ArrayList<>(); // Зберігаємо ВСЕ!
for (...) {
    allTriangles.add(triangle); // 166 млн записів!
}

// СТАЛО (новий код):
PriorityQueue<Triangle> topTriangles = new PriorityQueue<>(50000);
for (...) {
    if (queue.size() < 50000) {
        queue.add(triangle);
    } else if (better than worst) {
        queue.poll(); // Видаляємо гірший
        queue.add(triangle); // Додаємо кращий
    }
}
// Зберігаємо тільки ТОП 50,000!
```

### 2. **Обмеження розміру черги**
- Замість 166 млн трикутників зберігаємо тільки **топ 50,000**
- Економія пам'яті: з ~16 ГБ до **~5 МБ** (99.97% економії!)

### 3. **Adaptive progressUpdateInterval**
- Для малих наборів: оновлення кожні 100 трикутників
- Для великих наборів: оновлення кожні 100,000+ трикутників
- Менше викликів callback = швидше виконання

## 🎯 Чому все ще O(n³)?

**Неможливо уникнути O(n³)** для перебору всіх комбінацій:
```
for i in 0..n:
  for j in i+1..n:
    for k in j+1..n:
      // Перевіряємо трикутник (i,j,k)
```

Але **економимо на пам'яті** і **зменшуємо константу**!

## 💡 Автоматичні оптимізації:

Backtracking **автоматично переключається** на жадібний режим коли:
- Більше 100 точок, АБО
- Більше 1000 кандидатів-трикутників

Це гарантує що програма **завжди завершиться** за розумний час!
